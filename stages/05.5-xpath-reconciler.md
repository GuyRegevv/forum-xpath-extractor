# Stage 5.5 — XPath Reconciler

## Purpose

After Stage 5 generates and validates XPaths, compare the **match counts** across
all four fields. If any field matches fewer elements than the reference count, fire
a targeted LLM diagnostic call — providing the full context of the discrepancy and
the actual HTML of the problematic rows — and ask it to either produce a better XPath
or explain why the gap is intentional.

This stage adds an `explanation` field to the output for any field that was
investigated, and optionally replaces the XPath with a better one if the LLM
found one.

---

## Why a Reconciliation Step

Stage 5 validates that an XPath correctly identifies the target value. It does not
verify that the XPath is **complete** — that it matches every thread row on the page.

A field like `last_post_author` could be validated as `correct` with 18/20 matches
if the example thread happened to have a linked username. But 2 threads with deleted
accounts are invisible to that XPath. The pipeline would report success while silently
missing 10% of rows.

This stage closes that gap by:
1. Detecting count discrepancies automatically
2. Giving the LLM enough focused context to reason about *why* they exist
3. Letting the LLM decide whether a better XPath is possible, or whether the gap
   is structural and expected

The LLM is better positioned to make this judgment than any heuristic — because it
can read the actual HTML of the problem rows and understand the semantics.

---

## When It Fires

The reconciler fires **only when there is a discrepancy**.

**Reference count:** the maximum `match_count` across all four fields. This represents
the best available proxy for "how many thread rows are on this page." If all four fields
agree on a count (even if that count is 15), there is nothing to investigate.

**Trigger condition:** for each field, if `field.match_count < reference_count`, that
field is flagged for reconciliation.

Fields with `confidence == "failed"` (0 matches) are also passed to the reconciler —
they are an extreme case of the same problem.

If no field is flagged, this stage is a no-op and returns the XPath results unchanged.

| Input          | Type         | Description                                           |

| `raw_html`     | string       | Original rendered HTML from Stage 1                   |
| `url`          | string       | The original input URL — included for LLM context     |

---

## Output

Returns an updated `XPathResults` where reconciled fields may have:
- A **revised XPath** replacing the original (stored as `xpath`)
- The **original XPath** preserved in a new field `original_xpath`
- An **explanation** string describing what the LLM concluded
- An updated `match_count` if the revised XPath was validated and improved on the original

Fields that were not reconciled are returned unchanged, with `explanation` and
`original_xpath` as `None`.

Updated `FieldXPathResult` model:
```python
class FieldXPathResult(BaseModel):
    xpath: str
    sample_value: str
    confidence: str           # "correct" | "best_effort" | "failed"
    iterations: int
    match_count: int
    original_xpath: str | None = None   # set if XPath was revised
    explanation: str | None = None      # set if reconciliation ran
```

## The Focused HTML Excerpt

The key design decision in this stage is **what HTML to send the LLM**.

Sending the full raw HTML (140+ KB) would be wasteful and would dilute the
signal. Instead, extract only the thread rows that are **missing from the
flagged field's XPath** — the rows that the reference XPath (title) found but
the flagged field did not.

### How to extract problem rows

1. Run the **title XPath** at the element level (not text) to get the 20 title
   `<a>` elements
2. Walk up the DOM from each title element to find the **common ancestor row** —
   the nearest ancestor that contains all four fields (typically a `<div>` with
   a class like `structItem`)
3. Run the **flagged field's XPath** at the element level to identify which rows
   have a match and which do not
4. Collect the row elements that title found but the flagged field missed
5. Serialize those rows using `lxml.etree.tostring(..., encoding="unicode")` and
   include them in the prompt

This means the LLM sees only 1–3 rows of HTML, not the full page. The signal-to-noise
ratio is very high.

### Finding the row ancestor

Use lxml's `getparent()` to walk up from the title element until reaching a node
whose class contains a known structural marker. A reliable heuristic for XenForo
(and most forum software): walk up until the element has more than 2 child subtrees
with distinct class names (i.e., it is a container row, not a leaf).

Alternatively, use `lxml`'s `tree.getpath(element)` and extract the deepest common
XPath prefix shared between the title element and the sibling fields — that prefix
identifies the row container.

## LLM Configuration

### Temperature
`0` — diagnostic reasoning should be deterministic.

### Max tokens
`800` — the response includes reasoning + optional XPath + explanation.

### Response format
JSON only via `response_format={"type": "json_object"}`.

Response schema:
```json
{
  "explanation": "...",
  "revised_xpath": "..." | null
}
```

- `explanation`: always present. One to three sentences describing why the
  discrepancy exists and what was found in the problem rows.
- `revised_xpath`: a new XPath string if the LLM found a better one, otherwise `null`.
  If `null`, the original XPath is kept unchanged.

Pydantic model:
```python
class ReconcilerResponse(BaseModel):
    explanation: str
    revised_xpath: str | None
```

## Prompt Architecture

### System prompt

```
You are an XPath diagnostics expert. You are given a forum page where one XPath
selector is returning fewer matches than expected.

Your job is to:
1. Examine the HTML rows that the selector is missing
2. Explain in one to three sentences why those rows are not matched
3. If possible, provide a revised XPath that matches all rows

Rules for revised XPath:
- Do not hardcode specific text values
- Use contains(@class, 'value') for class matching
- Use structural attributes, not text predicates with specific values
- The XPath must generalise to any page of this forum
- If no reliable generalised XPath exists, set revised_xpath to null

Always respond in this exact JSON format:
{"explanation": "...", "revised_xpath": "..." or null}
```

### User message

```
Forum URL: {url}

Reference count: {reference_count} thread rows found by the title XPath.

Field under investigation: {field_name}
Current XPath: {field.xpath}
Current match count: {field.match_count} (missing {reference_count - field.match_count} rows)

All XPaths for context:
  title            ({title_count} matches): {title_xpath}
  last_post_author ({author_count} matches): {author_xpath}
  last_post_date   ({date_count} matches): {date_xpath}
  link             ({link_count} matches): {link_xpath}

HTML of the {n} rows that were matched by title but NOT by the current XPath:

{problem_rows_html}

Why does the current XPath miss these rows? Can you provide a better XPath?
```

## Reconciliation Logic

```
function reconcile_xpaths(xpath_results, raw_html, url):
    reference_count = max(field.match_count for field in xpath_results)

    if all fields have match_count == reference_count:
        return xpath_results unchanged

    for each field where field.match_count < reference_count:
        problem_rows = extract_problem_rows(
            raw_html,
            reference_xpath=xpath_results.title.xpath,
            flagged_xpath=field.xpath,
        )

        if problem_rows is empty:
            log warning, skip this field

        response = call_llm(
            field_name, field.xpath, field.match_count,
            reference_count, xpath_results, problem_rows_html, url
        )

        if response.revised_xpath is not None:
            new_count = run_xpath(response.revised_xpath, raw_html).count
            if new_count > field.match_count:
                field.original_xpath = field.xpath
                field.xpath = response.revised_xpath
                field.match_count = new_count
                log info: "[Reconciler] {field_name}: revised XPath improved count from {old} to {new}"
            else:
                log info: "[Reconciler] {field_name}: revised XPath did not improve count, keeping original"

        field.explanation = response.explanation

    return updated xpath_results
```

## Error Handling

| Scenario                              | Behaviour                                                                 |
|---------------------------------------|---------------------------------------------------------------------------|
| Problem rows extraction yields 0 rows | Log warning, skip reconciliation for this field, leave XPath unchanged    |
| LLM returns invalid JSON              | Log warning, set explanation to None, leave XPath unchanged               |
| LLM API call fails                    | Log warning, leave XPath unchanged — do not raise, this stage is optional |
| Revised XPath has syntax error        | Log warning, discard revision, keep original, still store explanation     |
| Revised XPath has lower match count   | Discard revision, keep original, still store explanation                  |
| All fields have equal match counts    | Return immediately — no LLM calls made                                    |

This stage must never raise exceptions or block the pipeline. It is an enhancement,
not a requirement. A failure here produces the same output as if the stage was skipped.

## What NOT to do

- Do not run reconciliation when all fields agree on count — even if that count
  seems low. Agreement means the XPaths are internally consistent.
- Do not accept a revised XPath that has a lower match count than the original
  — that is a regression, not an improvement.
- Do not send the full raw HTML to the LLM — extract only the problem rows.
- Do not raise exceptions — this stage is optional and must degrade gracefully.
- Do not replace an XPath that improved match count if the new count still does
  not reach reference count — partial improvement is still an improvement, accept it.
